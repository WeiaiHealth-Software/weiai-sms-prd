# 系统 UI 组件规范页面设计

## 背景

当前项目已经具备完整的后台骨架，包括：

- 左侧一级 / 二级菜单。
- 顶部 Header。
- 基于 `react-router` 的页面路由结构。
- 多个 CRM、门店、数据分析页面的卡片式内容区。

用户提供了另一套系统中的「系统 UI 组件规范」源码，位置为 `docs/UI 组件规范/`。该源码包含完整的页面结构与示例内容，但不能直接在当前项目中原样引用，原因如下：

- 依赖当前仓库不存在的路径别名，如 `@/components/...`、`@/store/...`。
- 依赖另一套系统的状态管理与 Header 标题写入逻辑。
- 组件目录位于 `docs/`，属于参考资料，不适合作为生产页面直接挂载。

因此，本次目标不是“原样复制运行”，而是“基于参考源码，在当前项目内按现有工程结构适配重建一个可运行页面”。

## 目标

在当前系统中新增一个独立一级菜单「系统 UI 组件规范」，点击后进入独立页面，页面内容依据 `docs/UI 组件规范/` 中的参考实现进行迁移与适配。

页面需要满足以下要求：

- 作为一级菜单存在。
- 不带子菜单。
- 使用当前项目既有布局骨架。
- 保留参考页的核心信息架构与示例内容。
- 适配当前项目的样式、路由与组件能力。

## 不做事项

本次范围不包含以下内容：

- 不直接把 `docs/UI 组件规范/components/` 目录作为运行时代码接入。
- 不引入另一套系统的状态管理方案。
- 不还原另一套系统的完整 Header 行为。
- 不要求与参考页逐像素一致。
- 不重构现有侧边栏整体样式，只新增入口并接入页面。

## 设计决策

### 方案选择

本次采用“方案 A：按当前系统风格适配重建”。

选择原因：

- 与当前项目结构兼容性最高。
- 开发成本低于完整像素级复刻。
- 后续维护成本更低。
- 页面仍能保留参考源码的大部分信息结构与展示价值。

### 页面入口

新增一个新的一级菜单项：

- 菜单名：`系统 UI 组件规范`
- 形式：一级菜单，无子菜单
- 点击行为：直接跳转到独立路由页面

菜单层级要求：

- 与 `仪表盘`、`门店管理`、`数据分析`、`客户档案（CRM）` 等同级。
- 不挂在任何现有一级菜单之下。

### 路由设计

新增独立页面路由，例如：

- `/ui-spec`

路由接入方式：

- 在主路由中注册独立页面组件。
- 不经过 `GenericPlaceholder`。
- 直接挂载到 `MainLayout` 的内容区。

## 页面结构

页面整体沿用当前项目的内容区风格，即：

- 顶部页面标题区。
- 卡片式内容模块。
- 内部使用 Tabs 切换子视图。

### 页面头部

页面头部应包含：

- 标题：`系统 UI 组件规范`
- 说明文案：概括该页用途，例如“统一沉淀后台系统常用 UI 规范与基础组件示例，争议以本页示例为准”。

头部风格遵循当前项目常用页面标题样式，不额外引入另一套系统的标题存储逻辑。

### Tabs 结构

页面内部保留 4 个页签：

1. `系统 UI 规范`
2. `系统基础组件`
3. `表格页`
4. `表单组件`

交互规则：

- 默认进入 `系统 UI 规范`
- 点击 Tab 后在页内切换内容
- 不新增子路由，保持单页切换

### Tab 一：系统 UI 规范

该页签用于承载整体设计原则，重点迁移参考源码中的：

- 主题色（brand）色板
- 副色（红 / 橙 / 绿 / 灰）色板
- 布局结构示意
- 菜单样式说明

展示形式：

- 顶部规则说明卡片
- 色板卡片
- 布局结构示意卡片
- 规则说明列表

### Tab 二：系统基础组件

该页签用于展示基础视觉语言与常见原子组件，重点保留：

- 图标规范
- 按钮样式
- 字体 / 文本规范
- 链接样式
- 标签（Tag）样式

展示形式：

- 各组件按章节分组
- 每个分组包含“说明 + 示例”
- 示例尽量用当前项目现有样式体系表达

### Tab 三：表格页

该页签用于展示后台表格页面标准布局，重点包括：

- 顶部规则说明
- 筛选工具栏
- 表格卡片
- 行内操作样式
- 删除确认弹窗示例

说明：

- 删除弹窗可沿用当前项目已有轻量交互模式实现。
- 重点在展示规范，不强调真实业务数据联动。

### Tab 四：表单组件

该页签用于展示输入类控件的统一风格，重点包括：

- Input
- Search Input
- Radio
- Checkbox
- Switch
- Select
- MultiSelect
- Textarea

说明：

- 若参考源码中的某些封装组件不能直接复用，则使用当前项目等价实现重建。
- 本页以演示交互和样式为主，不连接真实业务逻辑。

## 技术设计

### 文件边界

本次建议新增一个项目内页面文件，例如：

- `src/pages/ui/UiSpec.tsx`

同时修改以下现有文件：

- `src/config/menu.ts`
- `src/router.tsx`

如果页面体量较大，可视情况拆出少量局部展示组件，但首选先保持在单文件内完成，以便快速对齐参考内容。

### 参考源码使用方式

参考目录中的内容作为“设计素材”和“结构来源”使用，而不是直接 import。

迁移原则：

- 保留页面信息架构。
- 保留主要文案与示例组织方式。
- 将不兼容依赖替换为当前项目等价实现。
- 将样式改写为当前项目可运行的 Tailwind / React 写法。

### 样式策略

保持与当前项目统一：

- 内容区使用白底卡片、浅灰描边、圆角卡片。
- 文字层级与当前后台页面一致。
- Tabs 与交互按钮采用当前项目的主色体系。

允许保留参考页中的：

- 色板展示方式
- 规则说明块
- 示例布局

但不强求：

- 完整复刻另一套系统的 Header 和外层容器细节
- 完整复刻另一套系统的全局视觉语言

## 数据与交互

本页不依赖后端接口，全部使用本地静态示例数据。

主要交互包括：

- 切换 Tabs
- 表格页中的删除确认弹窗
- 表单示例中的本地输入交互

这些交互仅用于展示组件行为，不产生持久化结果。

## 验收标准

完成后需满足以下条件：

1. 左侧菜单新增一级入口 `系统 UI 组件规范`。
2. 该入口无子菜单，点击直接进入独立页面。
3. 页面包含 4 个页签：
   - `系统 UI 规范`
   - `系统基础组件`
   - `表格页`
   - `表单组件`
4. 页面主要内容已依据 `docs/UI 组件规范/` 进行迁移适配。
5. 页面可在当前项目中正常运行，不依赖参考工程中的别名和 store。
6. 不影响现有菜单与路由行为。

## 风险与控制

### 风险一：直接引用参考源码导致运行失败

控制方式：

- 明确不直接引用 `docs/UI 组件规范/` 目录中的运行时代码。
- 仅迁移内容和结构。

### 风险二：页面体量过大，单文件难维护

控制方式：

- 首版允许单文件实现。
- 若超过合理复杂度，再拆分局部展示组件。

### 风险三：页面风格与当前系统割裂

控制方式：

- 保持当前系统的卡片、字体、Tabs 和按钮风格。
- 只迁移参考页的信息结构，不强行复刻另一系统的全局外壳。
