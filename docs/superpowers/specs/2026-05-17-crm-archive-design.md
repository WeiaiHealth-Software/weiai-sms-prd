# CRM 客户档案改造设计

## 背景

当前客户档案模块的核心能力分为两部分：

- 查看档案：已支持就诊记录、配镜记录、预约记录、回访记录、消费记录。
- 新增档案：当前仅支持新增就诊记录，无法覆盖客户一次到店时的多项目统一录入场景。

本次改造目标有两个：

1. 在档案详情页新增“视光训练记录”查看能力。
2. 将“新增档案”从单一就诊表单升级为“客户基础信息 + 本次服务单 + 多模块录入”的完整页面。

## 现状

### 当前主系统

- `src/pages/crm/ClientDetail.tsx`
  - 已有详情页顶部客户信息区。
  - 已有 Tab：就诊记录、配镜记录、预约记录、回访记录、消费记录。
  - 就诊记录为完整详情模式，其余模块大多为表格或占位。
- `src/pages/crm/ClientVisitNew.tsx`
  - 当前是单页就诊录入表单。
  - 字段以医生、验光师、主诉、诊断、处理意见、复查信息为主。
- `src/pages/crm/mockData.ts`
  - 已有患者、就诊记录、预约记录、回访记录等 mock 数据。

### 原型来源

- `docs/CRM 客户档案管理系统/index.html`
  - 已实现“新增档案（支持一次录多项）”的 HTML 版逻辑。
  - 页面结构明确包含：客户基础信息、本次服务单、就诊档案、视光训练、配镜记录、收费详情。

## 目标

### 目标一：新增视光训练记录 Tab

在客户详情页的 Tab 中，于“就诊记录”后新增“视光训练记录”。

该 Tab 使用表格展示，至少包含以下字段：

- 视训时间
- 视训师
- 门店
- 备注
- 操作

操作列包含：

- 查看
- 删除

### 目标二：新增档案升级为多模块录入页

保留现有路由 `/crm/client/:id/visit/new`，但页面整体升级为完整录入页，用于承接客户一次到店的统一录入流程。

页面需要满足以下业务意图：

- 不再只录入就诊记录。
- 一次到店可选择录入就诊、视训、配镜、收费中的任意一个或多个模块。
- 默认保留“就诊档案”作为默认勾选模块。
- 支持只建档、不录服务单。

## 用户流程

### 查看档案页

1. 用户进入客户详情页。
2. 在顶部 Tab 中可切换到“视光训练记录”。
3. 用户可在表格中查看该客户历史视训记录。
4. 用户点击“查看”可查看单条记录详情。
5. 用户点击“删除”可移除该条记录，删除仅作用于本地 mock 状态。

### 新增档案页

1. 用户从客户详情页点击“新增档案”。
2. 进入新的多模块录入页。
3. 页面顶部展示客户基础信息与本次服务单快捷操作。
4. 用户选择本次服务涉及的模块。
5. 页面展示已选模块对应的录入表单。
6. 用户可点击“保存草稿”或“完成并返回”。
7. 本期保存行为先以 mock 交互闭环为主，不接真实接口。

## 页面结构设计

### 一、客户详情页

详情页维持当前整体框架，不调整顶部客户摘要区和原有主要布局。

#### Tab 顺序

Tab 顺序调整为：

1. 就诊记录
2. 视光训练记录
3. 配镜记录
4. 预约记录
5. 回访记录
6. 消费记录

#### 视光训练记录展示

展示形式：

- 使用与预约记录、回访记录一致的表格风格。
- 支持空状态展示“暂无数据”。

表格字段定义：

- `trainingTime`：视训时间
- `trainer`：视训师
- `store`：门店
- `note`：备注
- `actions`：查看、删除

查看交互：

- 本期优先使用页内轻量详情展示，不增加新路由。
- 可采用只读详情卡片或简易弹层风格。
- 内容以本期视训记录字段为主，不扩展复杂图表。

删除交互：

- 本期采用前端本地删除。
- 删除后更新当前页面列表状态。
- 暂不增加二次确认弹窗的复杂状态管理，可使用浏览器原生确认或简单按钮逻辑。

### 二、新增档案页

#### 顶部结构

页面顶部包含：

- 返回详情按钮
- 页面标题
- 页面说明文案
- 页面级操作按钮

按钮分为两层：

- 模块快捷按钮：完整录入、只建档
- 页面底部提交按钮：保存草稿、完成并返回

#### 客户基础信息区

展示客户已有身份信息，并支持录入本次建档所需基础字段。

建议字段：

- 姓名
- 手机号
- 性别
- 出生日期
- 客户来源
- 会员类型

说明：

- 本期保留前端输入与展示能力。
- 不做复杂校验策略。
- 既可展示已有客户信息，也可允许本地修改当前表单值。

#### 本次服务单区

用于选择本次到店实际涉及的业务模块。

模块项固定为：

- 就诊档案
- 视光训练
- 配镜记录
- 收费详情

交互规则：

- 默认勾选“就诊档案”。
- 用户可多选。
- 用户可只保留客户基础信息，不勾选任何服务模块。
- 页面根据勾选状态决定是否显示对应模块录入区。

快捷动作：

- “完整录入”会勾选四个模块。
- “只建档”会取消所有服务模块勾选，仅保留基础信息建档。

#### 模块录入区

页面下方按模块卡片依次展示。

每个卡片统一包含：

- 模块标题
- 模块说明
- 当前状态
- 添加录入或编辑区域

状态定义：

- 未录入
- 草稿
- 已录入

### 三、模块字段边界

#### 1. 就诊档案模块

迁移现有 `ClientVisitNew.tsx` 的核心字段，作为多模块页面中的第一块。

保留字段：

- 医生
- 验光师
- 眼别
- 症状
- 病程
- 描述
- 诊断
- 处理意见
- 复查周期
- 预计复查日期
- 提醒日期

设计原则：

- 优先复用现有就诊录入字段与样式习惯。
- 不一次性迁入所有复杂检查表格。
- 本期以“主诉、诊断、处理、复查建议”的主流程为主。

#### 2. 视光训练模块

该模块为本次新增重点之一。

首期字段包含：

- 训练项目
- 训练时长（分钟）
- 训练完成度（%）
- 训练备注

补充显示字段：

- 视训时间
- 视训师
- 门店

说明：

- 页面录入时以简化表单为主。
- 查看档案页的视训记录列表从该类 mock 数据中读取。

#### 3. 配镜记录模块

本期保留主结构和基础字段，避免完全占位。

建议字段：

- 处方类型
- 镜片/镜架信息
- 配镜参数或备注

本期要求：

- 允许录入基础内容。
- 不实现复杂专业参数矩阵。

#### 4. 收费详情模块

本期保留与原型一致的基础结构。

建议字段：

- 收费项目
- 金额
- 支付状态
- 收费备注

本期要求：

- 支持基础输入与展示。
- 不接真实收银逻辑。

## 数据设计

### 新增类型

在 `src/pages/crm/mockData.ts` 中新增视训记录数据结构。

建议新增类型：

```ts
type TrainingRecord = {
  id: string;
  patient: string;
  trainingTime: string;
  trainer: string;
  store: string;
  note: string;
  project?: string;
  duration?: string;
  completion?: string;
};
```

用途：

- 客户详情页“视光训练记录”Tab 列表展示。
- 新增档案页中的视训模块 mock 数据来源。

### 新增 mock 数据

新增一组 `trainingRecords`，并至少覆盖以下场景：

- 有视训记录的客户
- 无视训记录的客户
- 既有就诊又有视训的客户

### 页面本地状态

详情页：

- 维护 `trainingRecords` 的本地删除状态。
- 维护当前查看中的视训详情状态。

新增档案页：

- 维护基础信息表单状态。
- 维护模块勾选状态。
- 维护各模块录入内容状态。
- 维护各模块状态标签。
- 维护“已保存”提示状态。

## 交互规则

### 新增档案页交互

- 默认进入页面时，“就诊档案”为已选。
- 点击“完整录入”后，四个模块全部勾选并展示。
- 点击“只建档”后，隐藏所有模块录入区，仅保留客户基础信息。
- 模块被勾选后显示表单卡片；取消勾选后隐藏对应卡片。
- 保存草稿显示成功提示，不做真实持久化。
- 完成并返回显示成功提示后返回客户详情页。

### 详情页视训记录交互

- 列表数据按客户姓名筛选。
- 点击“查看”展示只读详情。
- 点击“删除”后立即从当前视图移除。

## 技术方案

### 文件改动范围

- `src/pages/crm/ClientDetail.tsx`
  - 新增 Tab。
  - 新增视训记录表格区域。
  - 新增视训查看/删除交互。
- `src/pages/crm/ClientVisitNew.tsx`
  - 从单就诊表单重构为多模块新增档案页。
- `src/pages/crm/mockData.ts`
  - 新增视训记录类型与 mock 数据。
  - 按需补充新增档案页面的默认种子数据。

### 组件策略

- 尽量在现有页面文件内做局部拆分。
- 若 `ClientVisitNew.tsx` 体积显著增大，可提炼同目录下的小型表单区块组件。
- 不引入新状态库，不引入表单库。

### 样式策略

- 延续当前 CRM 页面现有 Tailwind 风格。
- 保持与 `ClientDetail.tsx`、`ClientVisitNew.tsx` 的视觉一致性。
- 不追求 HTML 原型 1:1 像素还原，重点迁移结构、层级和交互。

## 本期不做

- 不接入真实后端接口。
- 不实现跨页面真实持久化。
- 不实现复杂字段联动和表单校验。
- 不新增独立的视训详情路由。
- 不实现收费模块与收银系统的真实联动。
- 不做复杂专业检查矩阵整体迁移。

## 风险与控制

### 风险一：新增档案页体量过大

控制方式：

- 优先复用现有就诊录入字段。
- 其余模块保持轻量字段集。
- 必要时拆出局部子组件。

### 风险二：视训记录与就诊记录关系不清

控制方式：

- 本期仅要求同一客户维度下的并列记录展示。
- 不建立复杂主从数据关系。

### 风险三：页面交互过多导致可读性下降

控制方式：

- 模块卡片统一结构。
- 保留明确的状态标签和按钮分区。
- 页面只保留最核心操作。

## 验收标准

满足以下条件即视为完成：

1. 客户详情页新增“视光训练记录”Tab。
2. 视训 Tab 可按客户展示表格数据。
3. 视训表格包含：视训时间、视训师、门店、备注、操作。
4. 视训表格支持“查看”和“删除”交互。
5. 新增档案页升级为多模块录入页。
6. 新增档案页包含客户基础信息区、本次服务单区、四个模块卡片区。
7. 默认勾选就诊档案，支持完整录入与只建档。
8. 页面支持保存草稿、完成并返回的 mock 闭环。

## 实现建议顺序

1. 先扩展 `mockData.ts`，补齐视训记录数据结构。
2. 再改造 `ClientDetail.tsx`，完成新 Tab 与表格交互。
3. 最后重构 `ClientVisitNew.tsx`，完成新增档案整页迁移。
4. 收尾时补充诊断检查和基础自测。
