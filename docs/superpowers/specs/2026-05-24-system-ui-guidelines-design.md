# 全系统 UI 规范总纲与表格页规则设计

## 背景

当前项目已经具备统一的后台框架，包括：

- 左侧一级 / 二级菜单。
- 顶部 Header 与面包屑。
- 主内容区卡片式布局。
- CRM、门店、系统规范等多个业务页面。

项目中已经存在一个可视化规范页 `系统 UI 组件规范`，用于展示颜色、组件、表格页、表单页等示例内容；同时，用户希望进一步沉淀一份当前系统自己的书面设计规范 `DESIGN.md`，作为后续页面开发的统一依据。

在最近的页面对齐过程中，已经出现一类明确偏差：

- `档案管理` 页面所使用的表格页工具栏结构，符合系统规范。
- `标签管理` 页面将“页面标题 + 描述”放进了表格卡片顶部工具栏区域，导致左侧区域被错误占用，不符合当前系统表格页规范。

因此，本次目标不是单独修一个页面，而是明确“全系统 UI 规范总纲”，并将其中的“表格页规则”落到：

- `标签管理` 页面
- `系统 UI 组件规范` 页面中的表格页说明
- 仓库根目录 `DESIGN.md`

## 目标

本次需要完成三个层面的统一：

1. 明确当前系统自己的 UI 规范总纲。
2. 明确“表格页”的布局与交互规则，并将其写入可视化规范页与文档。
3. 修正 `标签管理` 页面，使其符合该规范。

最终应形成两种规范载体：

- 可视化规范页：`系统 UI 组件规范`
- 书面规范文档：仓库根目录 `DESIGN.md`

## 不做事项

本次范围不包含以下内容：

- 不一次性重构所有列表页为通用工具栏组件。
- 不要求本次抽象出全局 `TableToolbar` 组件。
- 不重构全局 Header、Sidebar、路由结构。
- 不要求对所有历史页面逐页改造，只聚焦本次涉及的规范与页面。

## 设计决策

### 方案选择

本次采用“规范先行 + 页面同步 + 文档落地”的方案。

选择原因：

- 能同时解决当前页面不一致问题和后续缺少依据的问题。
- 可视化示例页与书面规范文档相互补充，便于设计对齐和开发执行。
- 范围可控，不会把本次任务扩展成全项目重构。

### 规范载体分工

本次定义两个规范载体，职责明确区分：

#### 1. `系统 UI 组件规范` 页面

作用：

- 用于“展示”规范。
- 通过页面示例表达颜色、组件、表格、表单的推荐写法。
- 适合作为开发时的视觉参考。

特点：

- 可交互。
- 强调视觉示例。
- 适合作为演示与对照页面。

#### 2. 根目录 `DESIGN.md`

作用：

- 用于“定义”规范。
- 作为当前系统统一设计规则总纲。
- 明确页面结构、工具栏规则、按钮层级、间距和组件职责。

特点：

- 书面化、结构化。
- 面向开发与后续页面扩展。
- 不承担示例展示职责，以规则描述为主。

## 当前系统的表格页规范

### 一、页面层级规则

页面标题属于页面顶部结构，而不是表格卡片内部结构。

统一规则：

- 页面标题显示在 Header / 面包屑下方的页面顶部区域。
- 表格卡片内部不重复放页面标题。
- 表格卡片的顶部第一行只承担筛选、搜索和主操作，不承担页面标题或说明文案。

这意味着：

- `标签管理` 卡片顶部当前的“标签管理 + 描述文案”属于错误结构。
- 正确做法应参考 `档案管理` 列表页。

### 二、表格工具栏规则

表格页中，表格上方工具栏统一拆分为左右两区：

#### 左侧区域

职责：

- 放置筛选项
- 放置搜索输入框
- 放置日期筛选
- 若当前页面没有筛选项，则保留为空白

禁止事项：

- 不放页面标题
- 不放页面描述
- 不放模块说明文字

#### 右侧区域

职责：

- 放置主操作与次操作
- 例如：`新增XX`、`导出`

规则：

- 主操作按钮靠右排列
- 若存在多个操作，按“次操作在左，主操作在右”组织

#### 左右分隔

统一规则：

- Action 区域左侧必须有一条竖线分隔
- 该竖线用于明确区分“筛选区”和“主操作区”

表现形式参考当前 `档案管理` 页面：

- 左侧是搜索、筛选、日期
- 右侧是 `重置`、`搜索`、竖线、`导出`、`新增档案`

### 三、无筛选场景的处理规则

如果某个表格页没有筛选项：

- 左侧区域仍然保留布局位置
- 可以为空白
- 不允许用页面标题或说明文案填充该区域

这样可以保证：

- 不同表格页的工具栏结构保持一致
- 用户的视觉预期稳定
- 后续扩展筛选项时不需要推翻布局

### 四、表格卡片结构规则

推荐结构统一如下：

1. 页面顶部标题区
2. 表格卡片
3. 卡片内工具栏
4. 表格主体
5. 分页区

卡片内部结构顺序固定：

1. 工具栏
2. 表头 / 表格内容
3. 分页区

不建议在卡片内部额外插入“标题说明区”作为第一层。

## 页面改造设计

### 一、`标签管理` 页面

当前问题：

- 卡片内部顶部左侧放了 `标签管理` 标题和说明文案。
- 筛选与操作没有形成“左筛选、右 Action”的标准结构。
- 工具栏结构与 `档案管理` 页面不一致。

改造目标：

- 移除卡片内标题与说明。
- 卡片顶部工具栏左侧只保留 `全部类型` 筛选。
- 卡片顶部工具栏右侧只保留 `新增标签` 操作。
- 在 Action 区左侧增加竖线分隔。
- 工具栏组织方式与 `档案管理` 对齐。

改造后预期：

- 页面标题只在 Header / 面包屑区域体现。
- 表格卡片顶部成为纯工具栏，不再混入标题说明。

### 二、`系统 UI 组件规范` 页面

当前问题：

- `表格页开发规范` 文案虽然提到了“筛选区、搜索区与右侧主操作建议放在同一张工具栏卡片中”，但还没有把“页面标题必须在 Header 顶部、工具栏左侧不能放标题说明、Action 左侧必须有竖线”明确写清楚。

改造目标：

- 在 `表格页开发规范` 中增加明确规则描述。
- 在表格页示例中体现以下结构：
  - 页面标题属于页面顶部
  - 表格卡片第一行是工具栏
  - 工具栏左筛选、右 Action
  - Action 左侧有竖线分隔
- 如果需要，可在说明文案中加入一条“错误示例提示”，明确说明不应把页面标题放进卡片工具栏左侧。

## `DESIGN.md` 文档设计

### 文件位置

- 仓库根目录：`DESIGN.md`

### 文档定位

`DESIGN.md` 作为当前系统统一设计规范总纲，用于沉淀当前项目真实采用的 UI 规则，而不是抽象的通用设计理论。

### 建议章节

建议包含以下章节：

1. 设计目标与适用范围
2. 页面整体结构规范
3. Header 与页面标题规范
4. 卡片容器规范
5. 表格页规范
6. 表单页规范
7. 按钮与操作层级规范
8. 标签 / 徽标 / 状态样式规范
9. 间距与分隔规范
10. 当前项目中的正确参考页
11. 常见错误写法

### 本次必须写入的关键规则

至少明确以下规则：

- 页面标题统一位于 Header / 页面顶部区域。
- 表格卡片内部不重复放页面标题。
- 表格工具栏左侧永远是筛选 / 搜索 / 留白。
- 表格工具栏右侧永远是 Action 区域。
- Action 区域左侧有竖线分隔。
- 无筛选时左侧保留空白，不使用标题补位。

## 参考页面与对照依据

### 正确参考页

- `档案管理`

其工具栏结构符合规范：

- 左侧为筛选与搜索
- 右侧为操作按钮
- 中间有竖线分隔

### 当前反例

- `标签管理`

其当前错误点：

- 标题进入表格卡片工具栏
- 左侧区域被标题说明占用
- 工具栏职责不纯

## 技术边界

本次不强制抽象通用组件，但代码实现应遵循以下原则：

- 优先复用当前项目已有页面结构模式。
- 保持 `档案管理` 与 `标签管理` 的工具栏布局逻辑一致。
- `UiSpec` 页面更新以说明规则和展示示例为主，不引入过度工程化封装。
- `DESIGN.md` 内容以当前项目真实规则为准，不照搬外部设计系统术语。

## 验收标准

完成后应满足：

1. `标签管理` 页面不再在表格卡片顶部显示页面标题与说明文案。
2. `标签管理` 工具栏结构变为“左筛选、右 Action”。
3. `标签管理` 的 Action 区左侧有竖线分隔。
4. `系统 UI 组件规范` 页面中的表格页说明明确写出该规则。
5. 仓库根目录新增 `DESIGN.md`。
6. `DESIGN.md` 明确沉淀当前系统的 UI 设计总纲，并包含表格页规范。

## 风险与控制

### 风险一：只改 `标签管理`，但规范仍未沉淀

控制方式：

- 同时更新 `UiSpec`
- 同时新增 `DESIGN.md`

### 风险二：文档写成抽象空话，无法指导开发

控制方式：

- 文档中引用当前系统真实页面作为正例与反例
- 规则描述使用“必须 / 不允许 / 建议”这种可执行语言

### 风险三：过度抽象成通用组件导致本次范围膨胀

控制方式：

- 本次只统一规则和页面表现
- 通用组件抽象不作为本次必做项
